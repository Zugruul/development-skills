# Design — cdx/E4: CI, documentation & compatibility matrix
Grounded in: SPEC-CODEX-COMPAT.md §10 (§10.1-§10.3), §6.1/§6.6 (Codex plugin/skill validators), §12 (additive-only, no forked tooling)

## Components (this task, CDX-040, covers only §10.1)
`.github/workflows/ci.yml` — gains a new `validate-codex` job, run alongside the existing `test` and `validate-manifests` jobs, on the same triggers (`push: [main]`, `pull_request`).
The new job invokes this repo's existing hermetic test sections `plugins/spec-workflow/tests/section-codex-plugin-json.sh` and `plugins/spec-workflow/tests/section-codex-skill-lint.sh` (built for CDX-003/#175) via `run-tests.sh --section codex-plugin-json,codex-skill-lint` — it does not duplicate their logic.

## Decisions
**No Codex CLI installation/authentication in CI.** The real Codex plugin validator (`~/.codex/skills/.system/plugin-creator/scripts/validate_plugin.py`) and skill linter (`.../skill-creator/scripts/quick_validate.py`) are shipped by the Codex CLI's own runtime, not by the npm/brew package alone — verified empirically: a fresh `codex` install with an unauthenticated `$HOME` does not populate `~/.codex/skills/.system/` on `--help`/`doctor`/`--version`. Populating it appears to require an authenticated session. Provisioning Codex CI credentials (a new secret) is a decision this task does not make unilaterally — flagged as a follow-up, not blocking.

**Existing SKIP-safe sections are reused, not forked.** Per the `skip-not-crash-on-missing-external-tool-dependency` invariant (CDX-003/#175): the test sections already gate on tool presence and print a visible SKIP rather than fail when the external validator is absent. The new CI job inherits this behavior honestly — on a GitHub-hosted runner (no Codex installed) it will SKIP visibly, not silently pass and not hard-fail. This matches §12's additive-only principle: the job's presence adds coverage wherever the tool IS available (a dev machine, a future CI image with Codex installed) without breaking or faking anything where it isn't yet.

**"Deliberately broken manifest fails it" is verified locally, matching CDX-003's own precedent.** CDX-003's DoD was "validator run recorded in the PR" — i.e., a human/dev with Codex installed locally runs the real validator and records the output, rather than CI depending on it. CDX-040 follows the same precedent: the acceptance test (push a test branch with a deliberately broken `.codex-plugin/plugin.json` or `SKILL.md` description, confirm the new local sections fail, revert) is run and recorded by the implementing dev agent locally, not asserted as a live GitHub Actions failure.

**Pre-existing CI shellcheck breakage is fixed in the same change.** `.github/workflows/ci.yml`'s existing `shellcheck` step runs `shellcheck plugins/*/scripts/*.sh plugins/*/tests/*.sh` — missing `-x` (follow sourced files) and missing `plugins/*/scripts/lib/*.sh` from scope, unlike this repo's own local gate (`shellcheck -x plugins/spec-workflow/scripts/*.sh plugins/spec-workflow/scripts/lib/*.sh plugins/spec-workflow/tests/*.sh`). Without `-x`, shellcheck can't see functions defined in a sourced helper, producing false SC2218 "used before defined" errors across several test files — the last 8 CI runs on `main` are all FAIL because of this, unrelated to Codex work. Since CDX-040's own DoD is "CI green on a real PR" and its acceptance requires "existing Claude jobs unchanged" (implicitly: unchanged AND passing, not unchanged-and-still-broken), aligning the CI shellcheck invocation with the already-correct local gate invocation is treated as in-scope for this task rather than a separate hotfix — it is a 2-line change to an existing step, not new functionality, and is required to make this task's own acceptance bar achievable.

## Out of scope for this task
§10.2 (README updates, CDX-041) and §10.3 (compatibility matrix, CDX-042) — separate, later E4 tasks.
Actually installing/authenticating the Codex CLI in CI — flagged as an open follow-up if a future task wants CI itself (not a local dev run) to catch a broken manifest.
