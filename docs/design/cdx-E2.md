# Design — cdx/E2: Host-aware delegation & model resolution
Grounded in: SPEC-CODEX-COMPAT.md §8.1-§8.3, §12 Invariants, §14 Testing strategy

## CDX-020 — additive `models.codex.capability` schema + normalizer (§8.1)

**§8.1 exact text**: "THE SYSTEM SHALL accept an optional `models.codex.capability` field (`fast`|`balanced`|`deep-review`|`large-context`) alongside each identity's existing Claude `models` list in `delegation.identities.<role>`, read only by the Codex adapter; omitting it SHALL NOT change Claude behavior or require any existing consumer repo to update its config."

**The structural problem**: `delegation.identities.<role>.models` is currently defined (schema + `identity_lib.py` + `validate-config.py`) as a flat array of Claude model-id strings — e.g. `models: [claude-sonnet-5, "claude-sonnet-5[1m]"]`. A single YAML/JSON key can hold only one shape at a time, so `models.codex.capability` cannot literally nest under the SAME key while `models` stays an array. The additive, zero-required-changes solution (per §12's own invariant) is to widen `models`'s accepted shape via `oneOf`: it may still be a flat array (100% of existing repos, unchanged, including this repo's own `project.yaml`), OR an object `{claude: [...], codex: {capability: <enum>}}` for repos that want to declare Codex behavior. The literal path `models.codex.capability` in spec prose describes the OBJECT form's shape, not a universal path that exists regardless of which form is used.

**Three places this touches, investigated individually (none skippable — each enforces or reads `models` independently today)**:
1. **`schemas/project-config.schema.json`** (`definitions.identity.properties.models`) — editor-hint schema (VSCode YAML extension autocomplete/hover). Currently `{"type": "array", "items": {"type": "string"}, ...}`. Becomes `oneOf: [<the existing array schema, unchanged>, <new object schema: {claude: array-of-string (optional), codex: {capability: enum, additionalProperties: false} (optional)}, additionalProperties: false>]`.
2. **`scripts/validate-config.py`** (~line 154-160, inside the `delegation.identities` loop) — the REAL enforcement (this is what `board.sh config` / gate.sh actually run; the JSON schema file is editor-only, never loaded by this validator). Currently rejects anything for `models` that isn't `isinstance(list) and all(isinstance(str))`. Must be extended: array form keeps its exact existing checks (including the shorthand-model-id rejection via `C._shorthand`, unchanged); object form validates `models.claude` with the SAME list/shorthand rules (reusing, not duplicating, `C._shorthand`), and `models.codex.capability` against the 4-value enum — an unrecognized capability value is a validation ERROR (`errs.append(...)`), never silently ignored or defaulted, matching the acceptance criterion's "fails clearly, doesn't silently default."
3. **`scripts/identity_lib.py`** — the actual RUNTIME resolution `identity.sh` (and `board.sh`, `implement-task`, etc.) call. `resolve_role()` currently does `it.get("models") or dflt.get("models") or []` and stuffs the raw list straight into `info["models"]`. **CDX-020 does NOT touch `resolve_role()`'s existing return shape or `identity.sh`'s bash-parsed `models:` output line — that's explicitly out of scope, reserved for CDX-021** (`implement-task`/`build-next` Codex model selection, a separate later task that WIRES this resolution into the actual dev/reviewer briefing steps). CDX-020 adds a new, STANDALONE, unit-testable function — `normalize_models(spec) -> (claude_models: list[str], codex_capability: str|None, error: str|None)` — that CDX-021 will import and call; it is not yet invoked from any live code path in this task, only exercised by its own unit tests. This keeps CDX-020's blast radius to exactly "schema is additive, a pure function exists and is correct," per its own 5-point acceptance scope; CDX-021 owns the actual behavior change to briefs.

**`normalize_models(spec)` contract**:
- `spec is None` (identity has no `models` key at all — legacy or role using role-default) → `([], None, None)`.
- `spec` is a `list` (today's universal form) → `(list(spec), None, None)` — Claude models pass through unchanged, no Codex capability.
- `spec` is a `dict` → `claude = spec.get("claude") or []` (not-a-list coerced to `[]`, matching validate-config.py's tolerance of a malformed shape being a VALIDATION error, not a normalizer crash — the normalizer is defensive, validation is where a bad shape gets reported); `cap = (spec.get("codex") or {}).get("capability")`; if `cap` is set and not one of `{fast, balanced, deep-review, large-context}` → return `(None, None, "<clear message naming the offending value and the valid set>")` — the SENTINEL for "fails clearly, doesn't silently default" (caller must check `error` before trusting the other two fields); otherwise `(claude, cap, None)`.
- Anything else (a string, a number — malformed config that validate-config.py should have already caught) → `([], None, None)`, same defensive-not-crashing stance as the malformed-claude-list case above.

**Constraints to preserve verbatim**: the array form's existing shorthand-rejection behavior (`C._shorthand`) in `validate-config.py` is UNCHANGED for the array form and REUSED (not reimplemented) for the object form's `models.claude` sub-list. This repo's own `project.yaml` (flat arrays for dev/reviewer) must validate with ZERO changes — this is the literal, directly-testable proof of "additive."

**Doc update** (DoD: "schema doc/comment updated if the schema file is user-facing" — it is, per the file's own JSON-schema `description` fields feeding VSCode hover text): the `models` property's `description` in the schema gains a sentence on the object form and the 4 valid `codex.capability` values, and `plugins/spec-workflow/README.md`'s identity-schema mention (search for where `delegation.identities`/`models` is documented in the giant Scripts/config paragraphs) gets a matching one-clause addition — dev agent's call whether an existing sentence can absorb it or a new clause is cleaner.

**New tests**:
- Schema: extend whatever section already lints `project-config.schema.json` (`section-schema-lint.sh`) with a case confirming the schema itself accepts BOTH forms for a synthetic identity (schema-level `oneOf` sanity, not full config validation).
- `validate-config.py`: extend `section-config.sh` (or wherever validate-config.py's existing `models` shorthand-rejection is tested) with cases: (a) legacy flat-array config (this repo's own `project.yaml` is the canonical fixture — must validate VALID, unmodified) — regression proof of additivity; (b) a config with `models: {claude: [...], codex: {capability: "balanced"}}` → VALID; (c) a config with an unrecognized `models.codex.capability` (e.g. `"super-fast"`) → INVALID, with a clear, specific error message naming the bad value.
- `identity_lib.py`: new unit tests (likely in `section-identity.sh`, wherever `identity_lib`/`identity.sh` already has coverage) for `normalize_models()` directly: `None` input, legacy list input, valid object-with-codex-capability input, and the unrecognized-capability-value input (asserting the error is returned, not silently swallowed into a default).

## CDX-021 — `implement-task`/`build-next` Codex model selection (§8.2, §8.3)

**§8.2 exact text**: "WHEN `implement-task`/`build-next` brief a dev or reviewer agent under Codex THE SYSTEM SHALL select a model via the resolved identity's `models.codex.capability` (falling back to a host-chosen default if unset) and SHALL NOT reference a Claude-only model id (e.g. `claude-sonnet-5[1m]`) in a Codex-run brief."

**§8.3 exact text**: "THE SYSTEM SHALL preserve, under both hosts: `covers`-glob identity routing, one-agent-per-task, independent developer/reviewer roles, role-prefixed agent naming (`dev-<task-id>`/`reviewer-<task-id>`), and per-commit author/committer attribution."

**What CDX-020 left unfinished (the reason this task exists)**: `normalize_models()` exists in `identity_lib.py` but is called from nowhere live. Two real, currently-Claude-only sites still assume the Claude `models` flat list unconditionally:
1. **`identity_lib.py`'s `resolve_role()`** (~line 87) — `it.get("models") or dflt.get("models") or []` is stuffed directly into `info["models"]`. Since CDX-020 widened `models` to ALSO accept the object form `{claude: [...], codex: {capability: ...}}`, an identity configured with the object form would make `resolve_role()` return a raw dict where callers expect a list — `identity.sh`'s own `", ".join(models)` (line 131) would silently iterate the dict's KEYS (`"claude"`, `"codex"`) instead of model ids, printing a nonsense `models: claude, codex` line. This is a LATENT BUG this task must fix, not a hypothetical.
2. **`identity.sh`'s own inline `render()` function** (~line 113-132, a SEPARATE, hand-duplicated read of `ident.get("models")` — does NOT call `resolve_role()` at all for this path) has the identical bug, independently.

**No host-detection mechanism exists yet anywhere in the plugin** (confirmed: `grep -rln "CODEX_HOST\|is_codex\|host.*codex" plugins/spec-workflow/scripts/` returns nothing). Every prior CDX task (§7.x) solved this by writing prose that the ACTING AGENT itself resolves at read-time (a Codex-side agent reading "delegate when the host supports delegation" already knows it's Codex, from its own runtime — no code needs to detect it). Model SELECTION is different: it needs a MACHINE-READABLE signal in `identity.sh`'s own output, because the orchestrator (of EITHER host) reads that output and picks a literal model id string to paste into a spawn call — capability language alone can't carry "pick from list X vs field Y," that decision needs an actual flag.

**Design decision: `identity.sh --host <claude|codex>`**, an explicit CLI flag (not an env var — a flag is directly testable per-invocation without mutating global process state, and every existing caller in this repo already invokes `identity.sh` as `bash ".../identity.sh" <role> [<path>]` with positional args only, so a NEW optional flag is trivially additive: omitting it preserves EXACT today's behavior, byte-for-byte).
- **Omitted (default, 100% of existing callers)** — unchanged: `render()` prints `models:` (the Claude list) exactly as today; no `codex-capability:` line ever appears. Zero behavior change for every existing test, skill, and script that doesn't pass `--host`.
- **`--host codex`** — `render()` NEVER prints a `models:` line (the hard "SHALL NOT reference a Claude-only model id" guarantee lives here, structurally, not by convention) and instead prints `codex-capability: <value>` when the identity's `models` declares one via the object form's `codex.capability`, or `codex-capability: (unset — host default)` when it's absent (§8.2's "falling back to a host-chosen default if unset" — the literal DEFAULT MODEL CHOICE is Codex's own runtime's job, not this plugin's; this plugin's job is only to never hand it a Claude id and to surface the capability hint when one is configured).
- **`--host claude`** — behaves identically to omitted (documented for symmetry/explicitness, not required).
- **Bad `models.codex.capability` value** (already rejected at config-validate time by CDX-020's `validate-config.py`, but `identity.sh` must ALSO fail clearly at resolve-time, since a config could theoretically reach this point unvalidated, e.g. hand-edited between a `board.sh config` check and a build-next run) — `render()` returns the `normalize_models()` error through its EXISTING `err` return slot (the same one already used for template-resolution failures), which the surrounding loop already surfaces via `warns.append(...)` and an `UNRESOLVED (...)` print — no new error-handling path needed, this is a straight reuse of code that already exists.

**Implementation**:
- `identity_lib.py`: `resolve_role()` calls `normalize_models(it.get("models") if "models" in it else dflt.get("models"))`; on error, return status `"invalid"` (new status value, alongside `ok`/`off`/`unresolved`/`unknown`) with the normalizer's error message; on success, `info["models"]` = the normalized Claude list (unchanged shape/contents for every existing caller) and `info["codexCapability"]` = the normalized capability (`str` or `None`).
- `identity.sh`: `render()` calls `I.normalize_models(...)` the same way (this function is a plain, cheap, dependency-free call — duplicating the ONE call site across `identity_lib.resolve_role()` and `identity.sh`'s own `render()` is acceptable, matching this file's existing pattern of `render()` independently re-deriving from `ident`/`dflt` rather than calling `resolve_role()`; do NOT refactor `render()` into calling `resolve_role()` in this task — that's a bigger, unrelated refactor, out of scope). Add the `--host` flag (parsed alongside the script's existing `--check` handling), thread it into `render()`, branch the `models:`/`codex-capability:` line exactly as designed above.
- `implement-task/SKILL.md` and `build-next/references/auto-review.md` (the two dev/reviewer spawn-briefing sites CDX-013 already made capability-language-aware for the SPAWN mechanism) — add ONE new sentence each, host-branching the MODEL SELECTION specifically: "Under Codex, resolve with `identity.sh --host codex <role> <path>` and pick the model from its `codex-capability:` line (or the host's own default if it reads `(unset — host default)`) — never a Claude-only model id. Under Claude Code, resolve normally (no `--host` flag) and pick from `models:`, as today." Placed immediately after the existing model-selection sentence in each file (implement-task's "The resolved `models:` line is that agent's ALLOWED set..."; auto-review.md's "...`model:` a suitable id from the reviewer identity's allowed set...").

**Constraints to preserve verbatim (§8.3, all already true today — this task's job is to NOT regress any of them, not to build them fresh)**: `covers`-glob routing (`I.select()`, untouched), one-agent-per-task (a skill-prose convention, untouched), independent dev/reviewer roles (untouched), role-prefixed naming `dev-<task-id>`/`pr-reviewer-<pr-number>` (untouched, CDX-013 already verified these survive), per-commit attribution (`on-behalf` mode, `render()`'s `name`/`email` output — untouched, this task only touches the `models:`/`codex-capability:` lines). Existing tests for all of these (`section-identity.sh` and others) must still pass UNMODIFIED — cite/confirm them running green, do not weaken or duplicate them.

**New tests** (the acceptance criterion's "Codex-path integration test (fixture config + fixture identity)"):
- `identity.sh --host codex <role> <path>` against a fixture config with `models: {claude: [...], codex: {capability: "balanced"}}` → output contains `codex-capability: balanced`, does NOT contain the literal Claude model id string(s) from the fixture's `claude` list anywhere in stdout.
- Same fixture but `models.codex` entirely absent (legacy array or object-with-no-codex-key) → `--host codex` output shows `codex-capability: (unset — host default)`, still zero Claude id strings.
- Fixture with an unrecognized `models.codex.capability` (bypassing config-validate, injected directly as a test fixture to exercise `identity.sh`'s own defensive resolve-time check) → `--host codex` produces the clear normalizer error via the existing `UNRESOLVED (...)`/`warns` path, exit code matching today's unresolved-role convention.
- Regression: `--host codex` does NOT change `covers`-glob selection, multi-identity listing, or `on-behalf` output — run a couple of EXISTING `section-identity.sh` cases again with `--host codex` appended and confirm name/email/covers output is identical, only the models line differs.
- Omitted `--host` (no flag at all) against the SAME fixtures → byte-identical to `identity.sh`'s pre-CDX-021 output — the core "zero behavior change for existing callers" proof.

## Out of scope for CDX-020
`identity.sh`'s actual `models:` output line, `resolve_role()`'s return shape, and any dev/reviewer briefing behavior change — all CDX-021 (`implement-task`/`build-next` Codex model selection, §8.2/§8.3), a separate later task in this same epic. CDX-020 ships a normalizer function that exists and is correct; CDX-021 is the one that calls it from a live path.
