## §7.7 board.sh rate-limit resilience — AMENDED (flush concurrency contract + adopt initial status)

§7.7 (task #77) specified the queue/flush mechanism but not what happens when two flushes overlap. Live incident (2026-07-08, ~06:13Z UTC): 15 ops queued; the neural-view server's periodic `board.sh list` auto-flush raced another flush — both snapshotted the queue file (`cp`), then truncated it (`: >`), then replayed independently. A `queue_append` landing in the snapshot-then-truncate gap of whichever flush hadn't yet captured it was silently wiped by that flush's truncate: 6 of 15 ops were applied, 9 vanished, and the queue was left empty with no trace of the loss.

THE SYSTEM SHALL serialize flush attempts and SHALL NOT lose an op appended while a flush is in progress:

1. **Mutual exclusion.** `_flush_queue` SHALL take an `mkdir`-based lockdir mutex (`.claude/board-queue.lock/`, next to the queue file) before touching the queue file. `mkdir` is atomic on every POSIX filesystem and is the bash-3.2-portable equivalent of `flock` (macOS bash has no `flock` builtin). A flush that cannot acquire the lock SHALL SKIP immediately — print a one-line message and return success — and SHALL NEVER block waiting for the lock, nor replay the queue concurrently with the lock holder. The lock-acquisition attempt SHALL happen before any check of whether the queue file is currently empty: another flush having already moved the queue file aside (see below) leaves it legitimately, momentarily empty, and an emptiness check run ahead of the lock attempt would misreport that "locked" state as "nothing to do."
2. **Stale-lock recovery.** A lockdir older than `BOARD_QUEUE_LOCK_TTL` (default 600s, mtime-based) SHALL be treated as abandoned (a previous flush crashed mid-flush) — broken with a warning, then re-acquired by the current flush. A crashed flusher SHALL NOT be able to wedge the queue forever.
3. **Lost-append prevention.** Flush SHALL NOT snapshot-then-truncate the queue file. It SHALL atomically move the queue file aside (`mv`, same filesystem — a single rename syscall, not a copy-then-clear with a gap in between) before replaying from the moved-aside copy. Any op that must be re-queued (a rate limit re-trips mid-replay) SHALL be appended to the LIVE queue path — not the moved-aside copy — since a concurrent `queue_append` may have grown the live path in the meantime; appending (not overwriting) preserves that growth.
4. `queue_append` itself SHALL continue to need no additional locking beyond `O_APPEND` semantics: single `>>`-appended JSON lines at these sizes and write rates do not interleave.
5. **Stale-requeue dedup (review round 2, BLOCKING).** Mutual exclusion and the aside-move alone do not fully close the loop: a rate-limited requeue appends the OLDER op to the TAIL of the live queue file, but a NEWER op for the same issue can already be sitting there — appended live while the first flush was mid-flight — ahead of it. Replaying in plain file order then applies the newer op first (the item's current value becomes its target) and the stale older op second; the existing `cur == target` skip guard does not catch this, because `cur` is now the newer op's target, not the value the stale op is comparing against, so the stale op re-applies and regresses the value. Before replaying, THE SYSTEM SHALL dedupe the moved-aside batch by `(op, issue)` for op kinds whose replay can regress a value (`move`, `prio`, `est`), keeping only the entry with the most-recently-enqueued `ts` (ties keep the later file position) and discarding the rest. `ts` SHALL NOT be rewritten across a requeue, so it always reflects original enqueue time — the ordering signal dedup relies on — not append time.

## adopt — AMENDED (initial status)

`item-add` (the underlying `gh` mutation both `add`'s add-finish step and `adopt` use to put an issue on the board) does not itself set a Status field value — a newly added item has no status until something explicitly sets one. `add`'s add-finish step already moves the item to `FIRST_STATUS` after adding it; `adopt` did not, so an adopted issue could land on the board showing `-` (no status) instead of `Backlog`. `_do_adopt` SHALL, after a successful `item-add` for a not-yet-visible issue, poll briefly for the item's visibility (same 3-attempt/0.3s-interval cap as add's add-finish step) and, if visible and not already at `FIRST_STATUS`, move it there — matching `add`'s behavior. If the item never becomes visible in that window, or the status-set call itself hits a rate limit, the adopt SHALL still report success (the issue IS on the board); a rate-limited status-set SHALL be queued as a follow-up `move` rather than failing the adopt.

## §9 Invariants — AMENDED

- (from #92) a flush that cannot take the lockdir mutex SKIPS rather than blocking or replaying concurrently; replay never truncates the live queue file without first atomically moving it aside, so a concurrent `queue_append` can never land in a snapshot-then-clear gap and be lost.
- (from #92) `adopt` leaves every adopted item with a real Status, not `-`.
- (from #92 round 2) a stale requeued `move`/`prio`/`est` for an issue that already has a newer entry for the same op+issue elsewhere in the batch is dropped before replay, never re-applied — file order alone (and the `cur == target` skip) is not sufficient to prevent a stale op from regressing a value a newer op already advanced past.
