## §3 Non-goals — AMENDED (clarify "cross-role brain reads")

§3 states "Cross-role brain reads — role privacy model is unchanged; `consult` remains
the only cross-role path" as a v1 non-goal. #163 (cross-identity correlation layer) does
NOT revisit that: it adds a derived, repo-level INDEX of which notes across roles are
about the same real-world entity — metadata about aboutness, never note content — built
purely from each note's own `entities:` frontmatter declaration. Reading a correlated
note still requires `consult` exactly as before; nothing new becomes readable that
couldn't be read before, only more *findable*. THE SYSTEM SHALL continue to treat direct
cross-role content reads as out of scope for v1; the entity index is explicitly not such
a read.

## §4 Glossary — AMENDED (new terms)

- **Entity** — a real-world thing (a card, a keyword, a hero, ...) that one or more
  notes, possibly in different roles' brains, are ABOUT. Referenced by a `kind:slug` key
  (e.g. `card:gone-in-a-flash`), where `slug` is a name-level kebab of the entity,
  deliberately independent of any note's own slug scheme.
- **Entity index** — `.claude/identities/entity-index.json`, a generated, committed,
  repo-level artifact mapping each entity key to the notes (across every role) that
  declare it and, where resolvable, an anchor note. Regenerated by `brain.py
  entity-index`, derived purely from note frontmatter, never hand-edited (same tier as
  `DIRECTORY.md`).
- **Anchor** — an entity's canonical fact note: the note in the entity kind's configured
  home role (`methodology.entityKinds`, kind -> role) when that role has exactly one note
  declaring the entity; `null` otherwise (ambiguous or no configured home role). Consumers
  drawing entity edges use an anchor<->member star, never all-pairs, to keep fan-out
  bounded for high-reference entities.

## §16 Cross-identity correlation layer (entity index) — NEW

Full design and resolved decisions: `docs/design/cross-identity-correlation.md`
(investigated against development-skills@a2cbcaf and fab-cli@main; open questions
resolved by the human 2026-07-12).

- **§16.1 declaration** — Notes MAY carry an optional `entities:` frontmatter list of
  `kind:slug` keys (see §4). `brain.py mint` SHALL accept `--entities "card:x,card:y"`;
  the field joins `KEY_ORDER` after `paths` and, like `tags`/`paths`, SHALL be re-passed
  on every re-mint to persist (mint does not carry old frontmatter forward beyond
  `strength`/`created`).
- **§16.2 generated index** — `brain.py <root> entity-index` SHALL scan every role's note
  frontmatter (never bodies, never links.json) and write
  `.claude/identities/entity-index.json`: `{"generated-by": "brain.py entity-index",
  "entities": {"<kind:slug>": {"anchor": "<role>/<slug>" | null, "notes": [["role",
  "slug"], ...]}}}`, with entity keys and note lists both sorted so regeneration is
  byte-for-byte diff-stable. A note whose file is a symlink (a kw-* physical/mirror pair)
  SHALL be attributed to its PHYSICAL home role only, never the role(s) it is mirrored
  into. An entity with only one declaring note anywhere SHALL still appear (the isolated-
  neuron case this layer exists to surface). The index is committed like `DIRECTORY.md`
  and regenerated by `entity-index`, by generator builders (consumer-repo concern, out of
  scope here), and at retro time.
- **§16.3 role privacy is unchanged** — Per-role `recall`/`query` SHALL NOT read the
  entity index, ever; their output for a note carrying `entities:` SHALL be byte-identical
  to a note without it. The index is a query-time join for whole-brain consumers only.
- **§16.4 ask-brain** — After its existing per-role recalls, `ask-brain` SHALL union the
  recalled notes' `entities:`, look each up in the entity index (regenerating it first if
  absent/stale), and for a correlated note in another role, `consult` it — consumer =
  the role whose recalled note triggered the correlation. This is the ONE sanctioned
  cross-boundary read path here, already logged/recurrence-counted by `consult` itself;
  answers remain attributed per role.
- **§16.5 ask-identity is awareness-only** — `ask-identity` MAY consult the entity index
  to STATE that another identity holds N notes about an entity and point the user at that
  identity or an explicit `consult`, but SHALL NEVER pull another brain's content into its
  own answer automatically. An identity learns from another only via an explicit ask (a
  human- or orchestrator-directed `consult`/`ask-identity` call) followed by a
  provenance-stamped mint (`learned-from`/`source-note`) into its own brain — never
  implicitly, never within one `ask-identity` turn. (Human decision, design doc §6.2 —
  this is a hard rule, not a default.)
- **§16.6 neural-view** — `build_graph` SHALL emit cross-role note-level edges
  `{"type": "entity", "entity": "<key>", "source": "<repo>/<roleA>/<slugA>", "target":
  "<repo>/<roleB>/<slugB>", "repo": <repo>}` per repo: an anchor<->member star when the
  entity has a resolved anchor, else a first-listed-note<->member star; same-role pairs
  SHALL NEVER be emitted (that remains `links.json`'s business). The index SHALL be read
  when `entity-index.json` is present; when absent, an equivalent map SHALL be derived
  live from frontmatter, so the view never requires a regen to show correlations. Nodes
  SHALL also carry their own `entities` list (like `tags`/`paths`), client-inspectable.
  The client SHALL exclude `type:"entity"` edges from the same-brain synapse physics/link
  layer exactly as it already excludes `type:"consult"` edges, and render them as a
  distinct low-opacity additive note-to-note line, colored by default as a gradient
  between the two endpoints' own brain colors (vertex colors), overridable per repo to a
  flat CSS color via `neuralView.entityEdgeColor` in `project.yaml` (default/absent =
  `"gradient"`). Hidden-brain filtering SHALL apply to entity edges exactly as it does to
  consult edges (both endpoints' brains must be visible).
- **§16.7 config surface** — `methodology.entityKinds` (object, kind -> home-role string)
  and `neuralView.entityEdgeColor` (string) are new, additive-only config keys; either or
  both absent SHALL preserve today's behavior exactly (no anchors resolved / gradient
  coloring). `neuralView` is a new top-level config section.
- **§16.8 links.json remains untouched by this layer** — Cross-identity keys are NEVER
  stored in a role's `links.json`: `cmd_prune` already treats any link whose target is
  absent from the role's own notes as `"target missing"` and deletes it, and `cmd_recall`
  fires/bumps `links.json` metadata during traversal — both would corrupt or destroy a
  cross-identity entry stored there. The entity index is a separate, regenerable artifact,
  analogous to `DIRECTORY.md`, never a `links.json` entry.

## §12 Invariants — AMENDED

- (from #163) the entity index is derived PURELY from note frontmatter — it can never be
  the only place an `entities:` declaration lives, and regenerating it can never lose a
  declaration that exists in a note (§16.2).
- (from #163) per-role `recall`/`query` output is unaffected by the presence or absence
  of `entities:` frontmatter or of `entity-index.json` (§16.3) — a hard regression gate,
  covered by a byte-identical-output test.
- (from #163) no content ever crosses a role boundary because of this layer without going
  through the existing, logged `consult` primitive (§16.3-§16.5) — the index shares
  metadata about aboutness, never note bodies.
