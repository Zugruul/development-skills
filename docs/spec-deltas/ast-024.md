## §7.7 — CLARIFIED ("flush in-flight turn state")

§7.7 reads: "WHEN the active assistant is switched THE SYSTEM SHALL flush in-flight
turn state, load the target assistant, and keep BOTH assistants' background work (task
queues, distillers) running throughout."

AST-024 implements this against what actually exists today (engine.py's `_select`,
turns.py, AST-010's worker registry) rather than a still-hypothetical async turn
pipeline. The turn pipeline (§8.1) is synchronous per HTTP request: `POST
/assistant/chat` runs load -> compose -> provider-call -> save entirely on the request
thread, under `_chat_lock_for(root)`, and returns before the next request is even
accepted. There is therefore no queued or in-progress turn living in engine memory
ACROSS requests for a switch to abandon. "Flush in-flight turn state" is accordingly a
DOCUMENTED NO-OP at v1, not an oversight — there is nothing to flush yet. What a switch
DOES actively do is reset the only cross-request state engine.py holds for "which
assistant is active" (`_selected`/`_gated`, AST-021/022) plus the new `_last_active`
bookkeeping this task adds.

Proposed EARS text, paste-ready for §7.7 once/if a queued or async turn pipeline lands
(E-later, not scoped here):

> §7.7 WHEN the active assistant is switched THE SYSTEM SHALL flush any in-flight or
> queued turn state held for the OUTGOING assistant (if the turn pipeline is ever made
> asynchronous — v1's synchronous per-request pipeline has none to flush), load the
> target assistant's selection state, and keep BOTH assistants' background work (task
> queues, distillers) running throughout, unaffected by the switch.

"Keep BOTH assistants' background work running throughout" already holds trivially
today: AST-010's worker threads (`self.workers`) are per-ENGINE, not per-assistant yet
(the distiller is E3, traces/tasks are E4/E6), and `_select` never reads or writes
`self.workers` — verified by a real assertion against the running thread registry
(section-assistant-switch.sh: "a switch never mutates the worker registry" / "both
assistants' workers stay alive across a switch").

## §7.8 — CLARIFIED ("sourced from its trace events")

§7.8 reads: "WHEN an assistant becomes active THE SYSTEM SHALL present an activation
digest of its background activity since last active (completed/failed tasks, minted
notes), sourced from its trace events."

`traces.sqlite` (§10.2) does not exist yet — it is E4. AST-024 builds the SAME response
shape from what exists per assistant repo today:

- **notes minted** — `<root>/.claude/brain-events.jsonl` (brain.py's `emit_event`,
  already a per-repo append-only JSON-per-line feed; `NoteMinted` events specifically),
  filtered by `role` and `ts` strictly after the assistant's last-active timestamp.
- **exchanges** — `<root>/.claude/assistant/session.jsonl` (store.py's `SessionStore`
  transcript), counted the same way.
- **tasks** — v1 ALWAYS returns `[]` with `tasksSource: "pending-E4"`. There is no
  per-assistant task-queue trace to read yet; this is a documented placeholder, never
  fabricated data.

`assistant/digest.py`'s `digest(root, since_ts, role="assistant")` is structured so E4
can swap `notesMinted`/`exchanges`/`tasks` to read `traces.sqlite` instead, without
changing the function's return shape (`{sinceTs, notesMinted, exchanges, tasks,
tasksSource}`) or any caller (`engine.py`'s `_select`, the template's
`renderAssistantDigest`).

Proposed EARS text, paste-ready for §7.8 once traces.sqlite (E4) lands:

> §7.8 WHEN an assistant becomes active THE SYSTEM SHALL present an activation digest of
> its background activity since last active (completed/failed tasks, minted notes),
> sourced from its trace events (`traces.sqlite`, §10.2) where available, falling back to
> `brain-events.jsonl`/`session.jsonl` for notes/exchanges until the per-assistant task
> trace exists.

## Bookkeeping this task adds (not a spec amendment, an implementation note)

`selection_store`'s persisted JSON gains an additive `lastActive: {name: isoTimestamp}`
field (§7.5's file, extended, never a new file) — the ISO-8601 moment each assistant
last STOPPED being the active one. `engine.py`'s `_select`, on a real switch (resolved
name differs from the previously selected one, and there WAS a previous selection):

1. stamps the OUTGOING assistant's `lastActive` entry to now;
2. builds the INCOMING assistant's digest from its OWN prior `lastActive` entry (before
   any stamp made in step 1) via `digest.digest`; no prior entry means "since the
   beginning of recorded history", never an error;
3. includes `digest` in the `POST /assistant/select` response ONLY on that real switch —
   an initial pick (nothing to switch FROM) or a same-name reselect (no change) return
   the plain AST-021/022 shape unchanged.
