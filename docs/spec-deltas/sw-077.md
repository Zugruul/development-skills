## §7.7 board.sh rate-limit resilience — ADDED

WHEN `board.sh move`, `prio`, `est`, `add`'s item-add step, or `adopt` encounters a GitHub rate-limit error THE SYSTEM SHALL append the operation to a durable local queue file (`.claude/board-queue.jsonl`, gitignored — one JSON object per line: `op`, its arguments, `ts`) instead of failing, print `QUEUED (rate-limited until <reset>): <op>`, and exit 0 so the build loop continues uninterrupted. The reset time SHALL be sourced from the REST `gh api rate_limit` endpoint, which remains available even when GraphQL itself is the thing exhausted.

`board.sh flush` SHALL replay the queue in order. A queued `move` SHALL be skipped (not reapplied) if the item's current status already matches the target, preventing a stale queued move from regressing a status that a newer, already-applied move has since advanced past. If a rate limit re-trips mid-replay, the current op and every op after it SHALL be re-queued verbatim (order preserved) and `flush` SHALL exit 0 — nothing is lost, nothing double-applies. Every board-READING command (`next`, `list`, `show`) SHALL attempt an automatic flush first whenever the queue file is non-empty, reporting what it flushed.

WHEN a READ operation (`next`, `list`, `show`) itself hits a rate-limited API THE SYSTEM SHALL fail fast with `RATE-LIMITED until <reset> — work continues; mutations queue; retry reads after reset.` — never a masked error (raw `gh` text, `board.sh`'s own "bad issue# or status", or an uncaught `JSONDecodeError` traceback from an ungated pipeline). `list`'s pipeline SHALL be gated on `gh`'s own exit code before any output is handed to a JSON parser.

`board.sh add`'s post-creation visibility poll (waiting for `gh project item-add` to become visible via `item-list`) SHALL be capped at 3 attempts (down from 10): past the cap, THE SYSTEM SHALL queue the remaining work (item-add if not yet done, then move to the first status, then priority) as an `add-finish` op and report `QUEUED (rate-limited until <reset>): item-add #<n>` instead of erroring — an unbounded retry loop was itself capable of burning quota back to zero.

`board.sh adopt <issue#>` (new subcommand) SHALL add an EXISTING issue to the board, reusing the same item-add path as `add`; a second `adopt` call on an issue already on the board SHALL be a no-op (idempotent); a rate limit during `adopt` SHALL queue it like any other mutation.

## §9 Invariants — ADDED

- `board.sh` never blocks the build loop on a GitHub rate limit: mutating operations queue locally and replay on the next board touch or an explicit `flush`; only a rate-limited READ fails the immediate call, and it fails with an explicit reset time, never a masked error.
- The board-queue file (`.claude/board-queue.jsonl`) is local, gitignored state — never committed, unlike the feedback archive.
