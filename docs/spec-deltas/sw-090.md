## §7.7 board.sh rate-limit resilience — AMENDED (detection is probe-based, not text-only)

§7.7 (task #77) specified rate-limit detection as a case-insensitive "rate limit" substring match on the captured `gh` output. Live production evidence (minutes after #77 merged, with the GraphQL quota exhausted) showed `gh` does not reliably surface that text: `gh project item-add`/`item-edit`/`item-list` instead failed with `unknown owner type` — no "rate limit" substring anywhere — so the text match never fired. `adopt` printed a raw `ERROR: ... gh project item-add failed: unknown owner type`, and `move` died in its item-id LOOKUP path with the pre-#77 `ERROR: bad issue# or status` before ever reaching the queue branch, because the lookup's own `gh` failure was silently discarded (`2>/dev/null`) rather than classified.

THE SYSTEM SHALL detect a GitHub rate limit in two layers, in order:

1. **Text fast path** (unchanged from §7.7): a case-insensitive `rate limit` substring in the captured output.
2. **Probe fallback (NEW)**: WHEN the text fast path does not match THE SYSTEM SHALL probe `gh api rate_limit` (the REST endpoint, which stays reachable even while GraphQL itself is exhausted) and read `.resources.graphql.remaining`. `remaining == 0` SHALL be treated as ground truth for rate-limited (queue the mutation / fail-fast the read, using the reset time from that same probe response). `remaining > 0` (or the probe itself failing/unparseable) SHALL NOT be treated as rate-limited — the original error SHALL be surfaced verbatim, never silently swallowed into the queue.

The reset-time reader SHALL prefer `.resources.graphql.reset` (the counter actually exhausted) and fall back to the top-level `.rate.reset` alias if `resources.graphql` is absent from the response.

WHEN `move`, `prio`, `est`, `add`'s add-finish step, or `adopt` first resolves an item-id (the LOOKUP, distinct from the mutation itself) AND that lookup's underlying `gh` call fails and is classified rate-limited (by either layer above) THE SYSTEM SHALL queue the operation (exit 0, `QUEUED` message) exactly as it would for a rate limit on the mutation call — NEVER surface `bad issue# or status` (or attempt a duplicate `item-add`) as a side effect of a masked, discarded lookup failure.

## §9 Invariants — AMENDED

- (from §7.7) restated: rate-limit detection is never solely a text match — every gh-project failure that isn't text-matched is probed against `gh api rate_limit`'s `resources.graphql.remaining` before being classified as either "queue it" or "a real error, surface it."
